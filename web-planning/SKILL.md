# Web Planning — 웹 프로젝트 기획 스킬

이 스킬의 목적: 사용자의 막연한 웹 개발 아이디어를 **빠짐없는 설계서**로 바꾼다.
코드는 한 줄도 작성하지 않는다. 오직 기획만 한다.

기획이 끝나면 설계서를 Markdown 파일로 저장하여, 이후 구현 단계에서 참조 문서로 활용한다.

---

## 기획 프로세스 개요

총 7단계로 진행한다. 각 단계를 순서대로 밟되, 사용자의 답변에 따라 유연하게 조절한다.

```
Step 1: 프로젝트 핵심 정의
Step 2: 사용자 역할과 권한 설계
Step 3: 페이지 구조 + 사용자 흐름 설계
Step 4: DB/데이터 구조 설계
Step 5: 디자인 방향 수립
Step 6: 놓치기 쉬운 영역 점검 (보안, SEO, 반응형, 에러처리)
Step 7: 최종 설계서 작성 + 구현 순서 확정
```

---

## Step 1: 프로젝트 핵심 정의

가장 먼저 "이 프로젝트가 뭔지"를 명확히 한다.
아래 항목을 대화를 통해 파악한다. 한꺼번에 묻지 말고 자연스럽게 진행한다.

### 파악할 항목

**1-1. 문제 정의**
- 지금 이 업무를 어떻게 처리하고 있는가? (엑셀? 수기? 기존 시스템?)
- 현재 방식의 가장 큰 불편함은?
- 이 웹을 만들면 구체적으로 뭐가 나아지는가?

**1-2. 핵심 기능 (최대 5개)**
- "이것만 되면 쓸 수 있다"는 기능은?
- 있으면 좋지만 나중에 추가해도 되는 기능은?
- 절대 필요 없는 기능은?

이 단계에서 핵심 기능을 **Must / Should / Could / Won't**로 분류한다:

| 구분 | 의미 | 예시 |
|------|------|------|
| Must | 없으면 못 씀 | 거래 목록 조회, 데이터 등록 |
| Should | 있어야 편함 | 검색 필터, 엑셀 다운로드 |
| Could | 있으면 좋음 | 대시보드 차트, 알림 |
| Won't | 이번엔 안 함 | 모바일 앱, 다국어 |

**1-3. 사용 환경**
- 사용자 수: 몇 명이 쓰는가?
- 기기: PC 위주? 모바일도?
- 네트워크: 사내망만? 외부 접속 필요?
- 기존 시스템 연동: 연결해야 할 시스템이 있는가?

---

## Step 2: 사용자 역할과 권한 설계

사내 관리 도구에서 가장 놓치기 쉬운 부분이 "누가 뭘 할 수 있는가"이다.

### 파악할 항목

**2-1. 사용자 유형 나열**
```
예시:
- 관리자 (Admin): 모든 권한
- 일반 사용자: 조회 + 등록
- 뷰어: 조회만 가능
```

**2-2. 권한 매트릭스**
각 사용자 유형별로 어떤 작업이 가능한지 표로 정리한다:

```
| 기능              | 관리자 | 일반사용자 | 뷰어 |
|-------------------|--------|-----------|------|
| 데이터 조회       | ✅     | ✅        | ✅   |
| 데이터 등록/수정  | ✅     | ✅        | ❌   |
| 데이터 삭제       | ✅     | ❌        | ❌   |
| 사용자 관리       | ✅     | ❌        | ❌   |
| 설정 변경         | ✅     | ❌        | ❌   |
```

**2-3. 인증 방식**
- 로그인 방식: 이메일+비밀번호? Google 로그인? 사내 SSO?
- 비밀번호 정책: 최소 길이, 특수문자 등
- 세션 관리: 자동 로그아웃 시간

사용자가 "권한 구분 필요 없어, 나만 쓸 거야"라고 하면 이 단계를 간소화해도 된다.
하지만 최소한 로그인 필요 여부는 확인한다.

## Step 3: 페이지 구조 + 사용자 흐름 설계

### 3-1. 페이지 목록 작성

모든 페이지를 나열하고 각 페이지의 역할을 한 줄로 정의한다:

```
페이지 구조 예시:

/login          — 로그인 페이지
/dashboard      — 메인 대시보드 (요약 지표 + 최근 활동)
/trades         — 거래 목록 (검색, 필터, 페이지네이션)
/trades/:id     — 거래 상세보기
/trades/new     — 거래 등록
/settings       — 설정 (사용자 관리, 시스템 설정)
```

### 3-2. 사용자 흐름 (User Flow) 설계

주요 업무 시나리오별로 사용자가 거치는 경로를 정리한다.
이 단계를 빠뜨리면 "페이지는 있는데 연결이 안 되는" 문제가 생긴다.

```
시나리오: 새 거래 등록

1. 로그인 → 대시보드
2. 사이드바에서 "거래 관리" 클릭 → 거래 목록
3. 우측 상단 "새 거래" 버튼 클릭 → 거래 등록 폼
4. 필수 정보 입력 → 저장 버튼
5. 확인 모달 → 확인 클릭
6. 거래 목록으로 돌아감 (성공 토스트 표시)
```

최소 3개의 핵심 시나리오를 작성한다:
- 가장 자주 하는 작업
- 데이터를 등록/수정하는 작업
- 데이터를 조회/검색하는 작업

### 3-3. 네비게이션 구조

사이드바 또는 상단 메뉴의 구조를 확정한다:

```
사이드바 메뉴 예시:

📊 대시보드
📋 거래 관리
  ├ 거래 목록
  └ 거래 등록
📄 신고서 관리
  ├ 수입 신고서
  └ 수출 신고서
⚙️ 설정
  ├ 사용자 관리
  └ 시스템 설정
```

## Step 4: DB/데이터 구조 설계

### 4-1. 핵심 데이터 엔티티 나열

다루는 데이터가 무엇인지 파악한다:

```
질문 예시:
- 이 시스템에서 관리하는 "것"들은 뭔가요? (거래, 고객, 제품, 문서 등)
- 각각 어떤 정보를 담고 있나요?
- 서로 어떤 관계가 있나요? (1:1? 1:다? 다:다?)
```

### 4-2. 데이터 모델 초안

각 엔티티의 주요 필드를 정리한다:

```
[거래 (Trade)]
- id: 고유 ID
- date: 거래일
- type: 수입/수출
- partner: 거래처명
- amount: 금액
- currency: 통화
- status: 상태 (진행중/완료/취소)
- created_at: 생성일
- updated_at: 수정일

[신고서 (Declaration)]
- id: 고유 ID
- trade_id: 연결된 거래 ID (FK)
- declaration_number: 신고번호
- ...
```

### 4-3. 데이터 관계도

엔티티 간 관계를 텍스트로 표현한다:

```
거래 1 --- N 신고서    (하나의 거래에 여러 신고서)
거래 N --- 1 거래처    (여러 거래가 하나의 거래처에)
사용자 1 --- N 거래    (한 사용자가 여러 거래 등록)
```

### 4-4. 저장소 결정

| 선택지 | 적합한 경우 |
|--------|------------|
| Firebase Firestore | 빠른 개발, 실시간 동기화 필요, 소규모 |
| Supabase (PostgreSQL) | 관계형 데이터, SQL 필요, 중규모 |
| 로컬 JSON/CSV | 프로토타입, 개인용 |

## Step 5: 디자인 방향 수립

### 5-1. 전체 톤 결정

```
질문:
- 깔끔하고 심플한 느낌? 화려하고 시각적인 느낌?
- 참고하고 싶은 사이트나 앱이 있는가?
- 밝은 테마? 어두운 테마? 둘 다 지원?
```

### 5-2. 컬러 시스템

최소한 아래 5가지 색상을 정한다:

```
- Primary: 주요 버튼, 강조 요소 (예: #2563EB 파란색)
- Secondary: 보조 요소 (예: #64748B 회색)
- Success: 성공/완료 상태 (예: #16A34A 초록)
- Warning: 경고/주의 (예: #EAB308 노랑)
- Danger: 에러/삭제 (예: #DC2626 빨강)
- Background: 배경색 (예: #F8FAFC 밝은 회색)
```

사용자가 잘 모르겠다고 하면 사내 관리 도구에 적합한 기본값을 제안한다.

### 5-3. 레이아웃 패턴

사내 관리 도구에서 가장 흔한 3가지 패턴:

```
패턴 A: 사이드바 고정 + 컨텐츠 영역 (가장 일반적)
┌──────┬─────────────────┐
│ 사이드 │    헤더          │
│ 바    │─────────────────│
│      │    컨텐츠         │
│      │                  │
└──────┴─────────────────┘

패턴 B: 상단 탭 네비게이션 (페이지 적을 때)
┌─────────────────────────┐
│ 헤더 + 네비게이션         │
│─────────────────────────│
│ 컨텐츠                   │
└─────────────────────────┘

패턴 C: 사이드바 접이식 (모바일 겸용)
┌──┬──────────────────────┐
│☰│    헤더               │
│  │──────────────────────│
│  │    컨텐츠             │
└──┴──────────────────────┘
```

### 5-4. 핵심 UI 컴포넌트 스타일

미리 정해두면 구현 시 일관성이 유지된다:

```
- 카드: border-radius, shadow 크기, padding
- 버튼: 크기 (sm/md/lg), 모서리 둥글기, hover 효과
- 테이블: 줄무늬 배경 여부, 행 높이, 헤더 스타일
- 입력 필드: border 스타일, focus 색상, 에러 표시 방식
- 모달: 배경 어둡기, 크기, 닫기 방식
```

### 5-5. UI 문구 규칙 정의

설계 단계에서 미리 문구 규칙을 정해두면 구현 시 일관성이 유지된다.
사용자와 아래 항목을 확인한다:

**버튼 문구 규칙:**
```
행동을 명확히 표현하는 동사형 사용:
✅ "저장", "삭제", "등록", "검색", "다운로드", "로그아웃"
❌ "확인", "OK", "Submit", "Go"

취소류: "취소" 통일 (닫기, 뒤로 등 혼용 금지)
위험 행동: "삭제", "초기화" 등은 빨간색 + 확인 모달
```

**라벨/안내문구 규칙:**
```
높임말 통일: "~합니다" 체 또는 "~해주세요" 체 중 하나 선택
✅ "거래를 등록하시겠습니까?" (통일)
❌ "거래를 등록할까요?" + "저장하시겠습니까?" (혼용)
```

**에러/성공 메시지 패턴:**
```
성공: "[대상]이(가) [행동]되었습니다."  예) "거래가 등록되었습니다."
에러: "[대상]을(를) [행동]하지 못했습니다." 예) "데이터를 불러오지 못했습니다."
확인: "[대상]을(를) [행동]하시겠습니까?" 예) "이 거래를 삭제하시겠습니까?"
빈 상태: "[대상]이(가) 없습니다. [행동 유도]" 예) "등록된 거래가 없습니다. 새 거래를 등록해보세요."
```

**한국어 맞춤법 주의사항:**
```
흔히 틀리는 것들:
- "되" vs "돼": "저장 되었습니다" ❌ → "저장되었습니다" ✅
- 띄어쓰기: "할 수 있습니다" ✅, "할수있습니다" ❌
- "데이터" 띄어쓰기: "데이터를" ✅, "데이타를" ❌
- "로그 인" ❌ → "로그인" ✅
- "다시한번" ❌ → "다시 한번" ✅
- "안 됩니다" ✅ vs "안됩니다" ❌
- "못 했습니다" ✅ vs "못했습니다" (둘 다 가능하나 통일)
```

이 규칙들은 설계서에 포함하고, 이후 CLAUDE.md에도 반영한다.

## Step 6: 놓치기 쉬운 영역 점검

이 단계는 사용자가 생각하지 못하는 부분을 Claude가 능동적으로 체크하는 단계다.
아래 4개 영역을 하나씩 점검하며 필요한 결정을 내린다.

### 6-1. 반응형/모바일 대응

```
점검 항목:
□ 모바일에서도 사용하는가? PC 전용인가?
□ 모바일에서 테이블이 잘 보이는가? (가로 스크롤? 카드형 변환?)
□ 사이드바는 모바일에서 어떻게 처리하는가? (햄버거 메뉴?)
□ 터치 대상 크기는 충분한가? (최소 44px)
□ 반응형 브레이크포인트: sm(640) / md(768) / lg(1024) / xl(1280)
```

각 페이지별로 모바일 레이아웃이 달라져야 하는 부분을 명시한다.

### 6-2. SEO (검색엔진 최적화)

사내 도구는 SEO가 덜 중요하지만, 외부 공개 페이지가 있다면 필요하다:

```
점검 항목:
□ 외부에 노출되는 페이지가 있는가?
□ 페이지별 title, description 메타태그
□ Open Graph 태그 (SNS 공유 시 미리보기)
□ sitemap.xml, robots.txt 필요 여부
□ URL 구조가 의미 있는가? (/trades/123 vs /?id=123)
```

사내 전용이라면: "SEO는 불필요, 대신 페이지 title만 적절히 설정" 으로 간소화.

### 6-3. 보안

사내 도구라도 반드시 점검해야 할 항목들:

```
필수 점검:
□ 인증: 로그인 없이 접근 가능한 페이지가 있는가?
□ 권한: 일반 사용자가 관리자 기능에 접근할 수 없는가?
□ 데이터 보호: 민감한 데이터(금액, 개인정보)가 노출되지 않는가?
□ API 보안: 인증 없이 API를 호출할 수 없는가?
□ 입력값 검증: SQL Injection, XSS 등 기본 공격 방어
□ HTTPS: SSL 인증서 적용 여부
□ 환경변수: API 키, DB 비밀번호가 코드에 하드코딩되지 않았는가?

추가 점검 (필요시):
□ 감사 로그: 누가 언제 무엇을 했는지 기록
□ 세션 타임아웃: 일정 시간 미사용 시 자동 로그아웃
□ IP 제한: 사내망에서만 접근 가능하도록 제한
□ 2단계 인증 (2FA) 필요 여부
```

### 6-4. 에러 처리 / 예외 상황

개발 후반에 발견되면 고치기 어려운 것들을 미리 정의한다:

```
점검 항목:
□ 네트워크 끊김 시 사용자에게 어떻게 알리는가?
□ 서버 에러(500) 시 보여줄 에러 페이지가 있는가?
□ 404 페이지 디자인
□ 폼 제출 실패 시 입력한 데이터가 유지되는가?
□ 동시 수정 충돌: 두 사람이 같은 데이터를 동시에 수정하면?
□ 대용량 데이터: 목록이 10,000건 이상이면 어떻게 처리하는가?
□ 로딩 상태: 데이터 불러오는 중에 뭘 보여주는가? (스켈레톤? 스피너?)
□ 빈 상태: 데이터가 아직 없을 때 뭘 보여주는가?
□ 타임아웃: API 응답이 오래 걸리면 몇 초 후에 에러 처리하는가?
```

**빈 상태/로딩/에러 3종 세트**: 모든 데이터 표시 페이지에 이 3가지를 정의해둔다:
```
거래 목록 페이지:
- 로딩 중: 테이블 스켈레톤 (5행)
- 데이터 없음: "등록된 거래가 없습니다. 새 거래를 등록해보세요." + 등록 버튼
- 에러: "데이터를 불러오지 못했습니다." + 다시 시도 버튼
```

→ 설계서 전체 템플릿은 `references/design-doc-template.md` 참조

### 설계서 작성 후 반드시 확인

사용자에게 설계서를 보여주고 아래를 묻는다:
1. "빠진 기능이나 페이지가 있나요?"
2. "우선순위(Must/Should/Could/Won't)가 맞나요?"
3. "이 순서대로 구현을 시작해도 괜찮을까요?"

승인을 받아야 기획 단계가 완료된다.

## 진행 시 주의사항

- 한 단계에서 질문은 최대 3개씩만 한다. 폭탄처럼 쏟아내지 않는다.
- 사용자가 "잘 모르겠다"고 하면 합리적인 기본값을 제안하고 확인받는다.
- 사용자가 "빨리 넘어가자"고 하면 해당 단계의 핵심만 빠르게 정리하고 넘어간다.
- 단, Step 1(핵심 정의)과 Step 3(페이지 구조)은 절대 건너뛰지 않는다.
- 기획 중 코드를 작성하고 싶은 유혹을 참는다. 이 스킬의 출력물은 오직 설계서다.
- 이전 대화에서 이미 파악된 정보는 다시 묻지 않고 활용한다.
