# Web Deploy — 최종 검수 및 배포 스킬

이 스킬의 목적: 완성된 프로젝트를 **배포 가능한 상태**로 만들고 실제 배포한다.
코드를 더 만드는 단계가 아니라, 기존 코드를 점검하고 다듬는 단계다.

---

## 진행 순서

```
Step 1: 기능 검수 (모든 기능이 동작하는가)
Step 2: UI/UX 검수 (보이는 것이 올바른가)
Step 3: 코드 정리 (불필요한 것 제거)
Step 4: 보안 점검
Step 5: 배포 설정 + 실제 배포
Step 6: 배포 후 확인
```

---

## Step 1: 기능 검수

설계서의 기능 목록을 기준으로 하나씩 체크한다.

### 1-1. Must 기능 전수 검사

```
설계서에서 Must로 분류된 모든 기능을 점검:

□ [기능1]: 정상 동작 확인
  - 정상 케이스: [입력] → [예상 결과]
  - 예외 케이스: [비정상 입력] → [에러 처리 확인]
□ [기능2]: ...
□ [기능3]: ...
```

### 1-2. 사용자 흐름 테스트

설계서에 정의된 핵심 시나리오를 처음부터 끝까지 따라간다:

```
시나리오 1: [가장 자주 하는 작업]
□ Step 1 → Step 2 → Step 3 → 완료
□ 중간에 끊기는 곳이 없는가
□ 완료 후 적절한 피드백(토스트)이 나오는가

시나리오 2: [데이터 등록]
□ 필수 필드 미입력 시 에러 표시되는가
□ 저장 성공 후 올바른 페이지로 이동하는가
□ 저장된 데이터가 목록에 바로 반영되는가
```

### 1-3. 엣지 케이스 점검

```
□ 데이터가 0건일 때 빈 상태 화면이 나오는가
□ 데이터가 1,000건 이상일 때 성능 문제 없는가
□ 매우 긴 텍스트 입력 시 레이아웃이 깨지지 않는가
□ 특수문자 입력 시 문제 없는가
□ 브라우저 뒤로가기 버튼 동작이 자연스러운가
□ 새로고침 시 현재 페이지가 유지되는가
□ 두 탭에서 동시에 같은 데이터 수정 시 어떻게 되는가
```

## Step 2: UI/UX 검수

### 2-1. 반응형 체크

3가지 화면 크기에서 모든 페이지를 확인한다:

```
□ 데스크톱 (1280px+): 정상 표시
□ 태블릿 (768px): 레이아웃 깨짐 없음
□ 모바일 (375px): 사용 가능한 수준

확인 항목:
□ 테이블이 모바일에서 볼 수 있는가 (스크롤 또는 카드형)
□ 모달이 모바일에서 화면을 벗어나지 않는가
□ 버튼/링크의 터치 영역이 충분한가 (44px+)
□ 입력 필드가 모바일에서 잘 보이는가
□ 사이드바 ↔ 햄버거 메뉴 전환이 자연스러운가
```

### 2-2. 디자인 일관성

```
□ 모든 페이지에서 같은 폰트 크기/색상 체계를 사용하는가
□ 버튼 스타일이 통일되어 있는가 (크기, 색상, hover)
□ 간격(padding, margin)이 일관적인가
□ 아이콘 스타일이 통일되어 있는가 (같은 라이브러리)
□ 성공/에러/경고 색상이 모든 곳에서 동일한가
□ 로딩 표시 방식이 통일되어 있는가 (스켈레톤 또는 스피너)
```

### 2-3. 접근성 기본

```
□ 이미지에 alt 텍스트가 있는가
□ 폼 필드에 label이 연결되어 있는가
□ 키보드만으로 주요 기능을 사용할 수 있는가 (Tab 이동)
□ 색상 대비가 충분한가 (텍스트가 잘 읽히는가)
□ 페이지 제목(title)이 각 페이지마다 적절한가
```

## Step 3: 코드 정리

배포 전 불필요한 코드를 제거하고 정리한다.

### 3-1. 기본 정리

```
□ console.log 제거 (개발용 로그 삭제)
□ 주석 처리된 코드 삭제 (의미 있는 주석은 유지)
□ 사용하지 않는 import 제거
□ 사용하지 않는 변수/함수 제거
□ TODO 주석 확인 (미완료 작업이 남아있는가)
□ 더미 데이터 제거 (실제 데이터로 교체 확인)
□ 하드코딩된 값이 없는가 (constants.ts로 이동)
□ TypeScript any 타입이 남아있지 않은가
```

### 3-2. 한국어 문구 최종 검수

모든 페이지를 돌면서 문구를 점검한다. 이 단계가 마지막 기회다.

```
[맞춤법 전수 검사]
□ 모든 버튼 텍스트가 동사형으로 통일되어 있는가
□ 같은 행동에 같은 단어를 쓰는가 (등록/추가/생성 혼용 없는가)
□ 높임말 체가 전체적으로 통일되어 있는가
□ "되" vs "돼" 오류가 없는가
□ 띄어쓰기 오류가 없는가
□ 조사(을/를, 이/가, 은/는)가 올바른가
□ 외래어 표기가 통일되어 있는가 (데이터/데이타, 메시지/메세지)

[메시지 패턴 통일]
□ 모든 성공 토스트가 같은 패턴인가
□ 모든 에러 메시지가 같은 패턴인가
□ 모든 확인 모달이 같은 패턴인가
□ 모든 빈 상태 문구가 같은 패턴인가
□ placeholder 텍스트가 일관적인가 ("검색어를 입력하세요" 통일)

[용어 통일 검사]
□ 같은 개념을 가리키는 단어가 프로젝트 전체에서 통일되어 있는가
  예) "거래처" / "파트너" / "업체" → 하나만 사용
  예) "목록" / "리스트" → 하나만 사용
  예) "등록" / "추가" / "생성" → 하나만 사용
```

### 3-3. 코드 품질 최종 검수

```
[변수명/구조]
□ 의미 불명확한 변수명이 남아있지 않은가 (data, temp, flag 등)
□ 200줄 이상인 컴포넌트가 있는가 → 분리 검토
□ 3회 이상 반복되는 코드가 있는가 → 함수/컴포넌트로 추출
□ 비즈니스 로직이 컴포넌트 안에 직접 있는가 → 커스텀 훅으로 분리

[주석]
□ 모든 주석이 한국어인가
□ 파일 상단에 역할 설명이 있는가
□ 코드를 그대로 번역한 무의미한 주석이 없는가
□ 오래된 주석이 현재 코드와 맞지 않는 경우가 없는가
```

### 빌드 테스트

```bash
# 빌드가 에러 없이 완료되는지 확인
npm run build

# 빌드된 결과물을 로컬에서 확인
npm run preview    # Vite
# 또는
npx serve dist     # 정적 파일 서버
```

빌드 에러가 있으면 모두 수정한다.

## Step 4: 보안 점검

설계서의 보안 체크리스트를 기반으로 최종 확인한다.

### 4-1. 필수 보안 항목

```
□ 환경변수 확인
  - .env.local에 민감 정보가 있는가 (코드에 하드코딩 안 됨)
  - .env.local이 .gitignore에 포함되어 있는가
  - .env.example이 존재하는가 (키 이름만, 값 없이)

□ 인증 확인
  - 로그인 없이 접근 가능한 페이지가 로그인 페이지뿐인가
  - 로그아웃 후 보호된 페이지에 접근 시 로그인으로 리다이렉트되는가
  - 세션/토큰 만료 시 적절히 처리되는가

□ 권한 확인 (권한 분리가 있는 경우)
  - 일반 사용자가 관리자 페이지 URL을 직접 입력해도 접근 불가한가
  - API 호출에도 권한 검증이 되는가

□ 입력값 검증
  - 모든 폼에서 서버 측 검증이 있는가 (클라이언트만 의존하지 않는가)
  - HTML 태그 입력이 그대로 렌더링되지 않는가 (XSS 방지)

□ 기타
  - HTTPS가 적용되어 있는가 (배포 환경에서)
  - 에러 메시지에 민감한 정보(스택 트레이스, DB 구조)가 노출되지 않는가
```

## Step 5: 배포 설정 + 실제 배포

### 5-1. 배포 환경 확인

CLAUDE.md 또는 설계서에서 배포 환경을 확인한다.

### 5-2. 환경별 배포

**Firebase Hosting:**
```bash
# Firebase CLI 설치 (최초 1회)
npm install -g firebase-tools

# 로그인
firebase login

# 초기 설정 (최초 1회)
firebase init hosting
# → public: dist
# → SPA: Yes
# → GitHub 자동 배포: 선택

# 빌드 + 배포
npm run build
firebase deploy --only hosting
```

firebase.json 확인:
```json
{
  "hosting": {
    "public": "dist",
    "ignore": ["firebase.json", "**/.*", "**/node_modules/**"],
    "rewrites": [
      { "source": "**", "destination": "/index.html" }
    ]
  }
}
```

**Vercel:**
```bash
npm install -g vercel
vercel login
vercel          # 최초 배포 (설정 안내)
vercel --prod   # 프로덕션 배포
```

**GitHub Pages:**
```bash
npm install -D gh-pages

# package.json에 추가
# "scripts": { "deploy": "npm run build && gh-pages -d dist" }

npm run deploy
```

### 5-3. 환경변수 설정

배포 환경에서도 환경변수를 설정해야 한다:

```
Firebase: firebase functions:config:set 또는 콘솔에서 설정
Vercel: 프로젝트 Settings → Environment Variables
Netlify: Site settings → Environment variables
```

## Step 6: 배포 후 확인

실제 배포된 URL에서 최종 확인한다.

```
□ 사이트가 정상적으로 열리는가
□ HTTPS가 적용되어 있는가 (자물쇠 아이콘)
□ 로그인이 정상 동작하는가
□ 데이터 조회/등록/수정/삭제가 되는가
□ 모바일에서도 정상 동작하는가
□ 이미지/폰트가 정상 로드되는가
□ 404 페이지가 올바르게 표시되는가
□ 브라우저 콘솔에 에러가 없는가
□ 페이지 로딩 속도가 적절한가 (3초 이내)
```

### 배포 완료 후 사용자에게 전달

```
✅ 배포 완료!

🔗 URL: https://[프로젝트].web.app (또는 해당 URL)

접속 정보:
- 관리자: [계정/방법]
- 일반 사용자: [계정/방법]

알려드릴 사항:
- [주의사항이나 제한사항]
- [추후 업데이트 계획]
```

## 주의사항

- 검수 없이 배포하지 않는다. 최소한 Step 1(기능 검수)은 반드시 거친다
- 배포 후에도 문제가 발견되면 즉시 수정 → 재배포한다
- 첫 배포 후 사용자가 실제로 사용해보면서 피드백을 받는 기간을 갖는다
- 환경변수를 배포 환경에 설정하는 것을 잊지 않는다
- 배포 URL과 접속 방법을 사용자에게 명확히 전달한다
