# Web Pages — 페이지 구현 스킬

이 스킬의 목적: 설계서의 구현 순서에 따라 페이지를 **하나씩** 만든다.
핵심 원칙: 한 번에 하나의 페이지만. 완료 → 확인 → 다음 페이지.

---

## 진행 원칙

### 구현 순서

설계서에 정의된 구현 순서(Step)를 따른다. 설계서가 없으면 아래 우선순위:

```
1. 대시보드 (메인 페이지, 전체 요약)
2. 핵심 데이터 목록 페이지 (가장 많이 쓰는 것부터)
3. 상세보기 페이지
4. 등록/수정 폼 페이지
5. 설정/관리 페이지
```

### 매 페이지 구현 시 따를 프로세스

```
1. 범위 선언: "지금부터 [페이지명]을 만듭니다"
2. 구조 설명: 이 페이지에 뭐가 들어가는지 사용자에게 설명
3. 구현: 코드 작성
4. 상태 3종 세트 포함: 로딩 / 빈 상태 / 에러 반드시 구현
5. 확인 요청: "확인해주세요. 수정할 부분 있나요?"
6. 수정 반영: 피드백 반영
7. 다음 페이지: 승인 후 다음으로
```

## 페이지 유형별 구현 가이드

### 유형 1: 목록 페이지 (List)

가장 흔한 페이지. 데이터를 테이블로 보여주고 검색/필터한다.

**반드시 포함:**

```
[상단 영역]
- 페이지 제목
- "새로 등록" 버튼 (우측)

[검색/필터 영역]
- 검색바 (이름, 번호 등 주요 필드)
- 필터: 날짜 범위, 상태, 카테고리 (설계서 기반)
- 필터 적용/초기화 버튼

[테이블]
- 컬럼: 설계서에 정의된 필드
- 정렬: 컬럼 헤더 클릭으로 오름/내림차순
- 행 클릭: 상세 페이지로 이동
- 페이지네이션: 한 페이지당 20건 (설정 가능)

[상태 3종 세트]
- 로딩: 테이블 스켈레톤 (5행)
- 빈 상태: "데이터가 없습니다" + 등록 유도 버튼
- 에러: "불러오기 실패" + 다시 시도 버튼
```

**자주 놓치는 것들:**
- 숫자 컬럼에 천단위 콤마
- 날짜 포맷 통일 (YYYY.MM.DD)
- 상태 컬럼에 뱃지 색상 (완료=초록, 진행중=파랑, 취소=빨강)
- 테이블이 비었을 때 빈 상태 화면
- 모바일에서 테이블 처리 (가로 스크롤 또는 카드형)

### 유형 2: 상세 페이지 (Detail)

목록에서 항목을 클릭했을 때 보이는 상세 정보 페이지.

**반드시 포함:**

```
[상단]
- 뒤로가기 버튼 (← 목록으로)
- 제목 (데이터 식별자)
- 수정/삭제 버튼

[본문]
- 정보를 섹션으로 구분 (기본 정보, 상세 정보, 관련 데이터)
- 라벨: 값 형태로 표시
- 관련 데이터 목록 (예: 거래에 연결된 신고서 목록)

[상태 3종 세트]
- 로딩: 섹션별 스켈레톤
- 데이터 없음: 404 페이지
- 에러: 재시도 버튼

[삭제 처리]
- 삭제 버튼 클릭 → 확인 모달 ("정말 삭제하시겠습니까?")
- 삭제 완료 → 목록으로 이동 + 성공 토스트
```

### 유형 3: 등록/수정 폼 (Form)

데이터를 입력하거나 수정하는 페이지. 등록과 수정은 같은 컴포넌트를 재사용한다.

**반드시 포함:**

```
[상단]
- 제목: "새 거래 등록" / "[거래명] 수정"
- 뒤로가기 버튼

[폼 본문]
- 필드를 섹션으로 구분
- 필수 필드 표시 (*)
- 적절한 입력 타입 사용:
  - text: 일반 텍스트
  - number: 숫자 (천단위 콤마 표시)
  - select: 선택형 (상태, 카테고리)
  - date: 날짜 선택
  - textarea: 긴 텍스트 (비고, 메모)

[유효성 검사]
- 필수 필드 미입력: 빨간 테두리 + 에러 메시지
- 형식 오류: 실시간 표시 (입력 중 또는 포커스 해제 시)
- 제출 시 전체 검증

[버튼]
- "저장" (primary) + "취소" (secondary)
- 저장 클릭 → 확인 모달 → 저장 처리
- 저장 중 버튼 비활성화 (중복 제출 방지)
- 저장 완료 → 목록 또는 상세 페이지로 이동 + 성공 토스트

[수정 모드 추가 사항]
- 기존 데이터를 폼에 미리 채워넣기
- 변경된 필드 감지 (변경 없으면 저장 버튼 비활성화)
- 수정 중 페이지 이탈 경고 ("저장하지 않은 변경 사항이 있습니다")
```

**자주 놓치는 것들:**
- 폼 제출 실패 시 입력 데이터 유지
- 저장 중 로딩 상태 표시
- 중복 제출 방지
- 수정 시 변경사항 없는데 저장 누르는 경우

### 유형 4: 대시보드 (Dashboard)

프로젝트의 메인 화면. 핵심 지표를 한눈에 보여준다.

**반드시 포함:**

```
[요약 카드] (상단 가로 배치)
- 3~5개의 핵심 지표
- 각 카드: 아이콘, 수치, 라벨
- 선택: 전월/전일 대비 증감 표시 (▲ 5%)

[차트 영역] (선택)
- 추이 차트 (라인/바): 월별, 주별 변화
- 분포 차트 (파이/도넛): 카테고리별 비율
- 라이브러리: recharts 추천

[최근 활동 / 할 일]
- 최근 등록/수정된 데이터 5~10건
- "전체보기" 링크 → 해당 목록 페이지로 이동

[반응형]
- 카드: lg 4열 → md 2열 → sm 1열
- 차트: lg 2열 → md 이하 1열
```

## 공통 UI 패턴

모든 페이지에서 일관되게 사용할 패턴:

### 토스트 알림
```
성공: 초록 배경, 3초 후 자동 사라짐
에러: 빨강 배경, 수동으로 닫기
정보: 파랑 배경, 3초 후 자동 사라짐
```

### 확인 모달
```
제목: "정말 삭제하시겠습니까?"
내용: "[항목명]을 삭제하면 복구할 수 없습니다."
버튼: "삭제" (빨강) + "취소" (회색)
```

### 페이지네이션
```
이전/다음 버튼 + 페이지 번호
현재 페이지 하이라이트
총 건수 표시: "전체 234건 중 1-20건"
```

## 구현 시 규칙

1. **기존 코드 참조**: 새 페이지를 만들 때 이미 만든 페이지의 스타일을 참조한다
2. **컴포넌트 재사용**: 테이블, 검색바, 페이지네이션은 공통 컴포넌트로 분리한다
3. **하드코딩 금지**: 상태값, 라벨 등은 constants.ts에서 관리한다
4. **타입 안전**: 모든 데이터에 TypeScript 타입을 적용한다
5. **상태 3종 세트**: 로딩/빈/에러 상태를 반드시 구현한다
6. **포맷 유틸 사용**: 날짜, 숫자, 통화는 utils.ts의 포맷 함수를 사용한다

---

## 주의사항

- 절대 두 페이지를 동시에 만들지 않는다
- 페이지가 복잡하면 페이지 내부도 분리한다 (검색 영역, 테이블 영역, 모달 등)
- 더미 데이터로 먼저 UI를 완성하고, 이후 실제 API/DB 연결은 별도 작업으로 한다
- 매 페이지 완료 후 반드시 사용자 확인을 받고 다음으로 넘어간다

## 페이지 구현 시 문구/코드 품질 규칙

매 페이지를 만들 때 CLAUDE.md에 정의된 규칙을 적용한다. 아래는 페이지 구현에서 특히 중요한 항목이다.

### UI 텍스트 체크리스트

페이지 구현 완료 후 아래를 점검한다:

```
[버튼]
□ 동사형으로 작성했는가 ("저장", "삭제", "등록")
□ 프로젝트 내에서 같은 행동에 같은 단어를 쓰는가
  - 새 데이터 추가: "등록" 통일 ("추가", "생성", "만들기" 혼용 ❌)
  - 데이터 변경: "수정" 통일 ("편집", "변경", "업데이트" 혼용 ❌)
  - 데이터 제거: "삭제" 통일 ("제거", "지우기" 혼용 ❌)

[라벨/텍스트]
□ 높임말 체가 통일되어 있는가
□ 같은 데이터를 가리키는 용어가 통일되어 있는가
  - 예) "거래처" vs "파트너" vs "업체" → 하나로 통일
□ 테이블 헤더가 간결한가 (긴 설명 대신 짧은 명사)

[메시지 패턴]
□ 성공 토스트: "[대상]이(가) [행동]되었습니다."
□ 에러 토스트: "[대상]을(를) [행동]하지 못했습니다. 다시 시도해주세요."
□ 확인 모달: "[대상]을(를) [행동]하시겠습니까?"
□ 빈 상태: "[대상]이(가) 없습니다. [행동 유도]"

[맞춤법]
□ "되" vs "돼" 올바르게 사용했는가
  - "저장되었습니다" ✅ / "저장 되었습니다" ❌
  - "수정됩니다" ✅ / "수정 됩니다" ❌
□ 띄어쓰기가 올바른가
  - "할 수 있습니다" ✅ / "할수있습니다" ❌
  - "로그인" ✅ / "로그 인" ❌
  - "불러오기" ✅ / "불러 오기" ❌
□ 조사가 올바른가
  - "거래가" (받침 없음+가) / "데이터를" (받침 있음+을)
  - "을/를", "이/가", "은/는" 앞 글자 받침에 따라 구분
□ 외래어 표기가 통일되어 있는가
  - "데이터" ✅ / "데이타" ❌
  - "콘텐츠" ✅ / "컨텐츠" ❌  
  - "메시지" ✅ / "메세지" ❌
```

### 코드 품질 체크리스트

```
[변수명/함수명]
□ 의미가 명확한 이름인가
  - ✅ filteredTrades, handleDeleteTrade, isFormValid
  - ❌ data, fn1, temp, flag
□ boolean은 is/has/can 접두사를 사용했는가
□ 이벤트 핸들러는 handle 접두사를 사용했는가
  - ✅ handleSubmit, handleSearch, handleDelete
  - ❌ submit, onClickBtn, doSave

[구조]
□ 한 컴포넌트가 200줄을 넘지 않는가 (넘으면 분리)
□ 같은 코드가 3회 이상 반복되지 않는가 (반복되면 함수/컴포넌트로 추출)
□ 비즈니스 로직과 UI 로직이 분리되어 있는가 (커스텀 훅 활용)
□ 하드코딩된 문자열이 없는가 (constants.ts로 이동)

[주석]
□ 한국어로 작성했는가
□ 파일 상단에 역할 설명이 있는가
□ 복잡한 로직에 "왜" 이렇게 했는지 설명이 있는가
□ 불필요한 주석 (코드를 그대로 번역한 주석)은 없는가
  - ❌ // 카운트를 1 증가시킨다
  - ✅ // 페이지네이션을 위해 현재 페이지 추적

[타입]
□ any 타입이 없는가
□ 컴포넌트 props에 interface/type이 정의되어 있는가
□ API 응답 데이터에 타입이 적용되어 있는가
```
